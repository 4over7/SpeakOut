# 词汇增强功能需求文档

**记录日期**：2026-03-05
**状态**：待设计/实现

---

## 背景与动机

当前 ASR 输出存在两类问题：
1. **音近字错误**：专业词汇发音相近但字形不同（如"科博斯" → Kubernetes）
2. **方言口音**：南方用户平翘舌不分（z/zh、c/ch、s/sh 混淆），导致识别偏差
3. **行业词汇缺失**：模型对低频专业词汇的上下文约束不足

---

## 核心设计原则

**不做硬替换**。不能因为匹配到音近词就强制替换——用户说的可能就是原词本身。必须基于概率判断：在当前上下文中，替换后的句子是否比原句更合理。

---

## 技术方案：三层概率打分

### 层一：声学置信度（Acoustic Confidence）
- Sherpa-ONNX 已输出每个 token 的声学置信分
- 只有置信度低于阈值的词才进入候选替换流程
- 高置信词不动（用户说的就是这个词）

### 层二：音近距离（Phonetic Distance）
- 将 ASR 输出词和词汇表候选词都转为拼音
- 计算拼音编辑距离（含平翘舌规则：z↔zh、c↔ch、s↔sh 视为距离 0）
- 距离超过阈值的候选词排除

### 层三：N-gram 语言模型上下文得分
- 使用 KenLM（C++ 轻量级 n-gram LM，通过 FFI 调用）
- 对每个候选替换计算：P(句子含候选词) vs P(句子含原词)
- 选择上下文概率更高的版本
- 按行业加载不同的 trigram 模型

**只有三层都满足时才替换**：声学不确定 + 音近 + 上下文更合理。

---

## 词汇来源

### 用户自定义词汇
- 用户手动输入词条（支持"错误写法 → 正确写法"对，或仅输入正确词）
- 系统自动生成拼音，用于音近匹配
- 存储在 SharedPreferences / 本地文件

### 行业预制词汇包
每个行业包含：
- 词汇对列表（常见误识别 → 正确词）
- 行业 trigram 语言模型（KenLM 格式，~5-15MB）
- 行业语料来源：Wikipedia 对应分类 + 公开技术文档

**预制行业**：
| ID | 名称 | 典型词汇示例 |
|----|------|-------------|
| `tech` | 软件/IT | Kubernetes、Docker、API、Git、Flutter |
| `medical` | 医疗 | 对乙酰氨基酚、腹腔镜、心肌梗死 |
| `legal` | 法律 | 原告、被告、仲裁、不可抗力 |
| `finance` | 金融 | 资产负债表、ETF、期权、做市商 |
| `education` | 教育 | 教案、学籍、核心素养 |

---

## 方言支持

平翘舌规则作为音近距离计算的特殊规则：
- z ↔ zh 编辑距离 = 0（不算距离）
- c ↔ ch 编辑距离 = 0
- s ↔ sh 编辑距离 = 0
- n ↔ l 编辑距离 = 0（部分南方口音）
- f ↔ hu 编辑距离 = 1（部分方言）

---

## 适用范围

- **所有 ASR 模型均适用**（纯后处理，不干预解码过程）
- 无需 LLM，完全离线
- 不影响现有 LLM 纠错功能（两者独立，可叠加）

---

## 关键技术组件

| 组件 | 方案 | 备注 |
|------|------|------|
| 声学置信度 | Sherpa-ONNX 现有输出 | 需确认 API 是否已暴露 |
| 拼音转换 | 打包轻量拼音库（C 层） | ~1MB |
| 音近距离 | 自定义编辑距离（含方言规则） | Dart 实现 |
| N-gram LM | KenLM，FFI 调用 | 每行业 5-15MB |
| 行业语料 | 需整理/训练 | 离线预制 |


## 实现阶段规划

- **Phase 1**：简单替换表（无概率，先上线解决 80% 问题）
- **Phase 2**：接入声学置信度 + 拼音音近匹配
- **Phase 3**：集成 KenLM，实现完整三层打分
- **Phase 4**：行业语料整理 + trigram 模型训练/打包
