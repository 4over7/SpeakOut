# 对《评审回复》的复评意见

> 日期：2026-03-19
> 身份：独立第三方技术团队
> 针对文件：`docs/review/response_to_review_2026_03_19.md`

## 一、结论摘要

这份回复整体是专业的、坦诚的，覆盖面也基本完整。我认可它对大多数核心问题的采纳态度，尤其是：

- 凭证安全存储问题
- LLMService 记录原始文本的问题
- Gateway 幂等与计费语义风险
- 版本信息多处复制源的问题

但从“独立第三方复评”的标准看，这份回复仍有两处表述需要收敛，否则容易给外部读者造成“整改已开始甚至已落地”的误解。

我的总体判断是：

**回复方向正确，专业度合格，但部分措辞偏满，建议先修正文档口径，再进入实际代码整改。**

## 二、我认可的部分

### 1. 没有回避核心事实

这份回复没有通过“语境解释”去否认问题本身，而是直接承认了以下事实：

- 凭证当前确实仍落在 SharedPreferences
- LLMService 确实会无条件写 `/tmp/SpeakOut.log`
- ASR provider 注册表与运行时实现存在落差
- Gateway 的 `/report`、`/redeem` 存在接口契约与一致性风险
- 版本号与工程指标存在失配

这一点是专业的，也是我认可的。

### 2. 技术判断大体靠谱

回复中关于以下几点的判断，我认为成立：

- macOS 上 `flutter_secure_storage` 对接 Keychain，适合作为敏感凭证存储方案
- 普通偏好配置与敏感凭证应分层存储
- LLM 原始文本日志必须纳入统一日志治理
- Gateway 后续应转向幂等化、可审计的计费接口

### 3. 优先级排序合理

把工作重点放在：

1. 安全存储与日志治理  
2. 能力声明与运行时实现对齐  
3. Gateway 幂等、计费与版本一致性  

这个排序我认可。

## 三、我不完全认可的部分

### 1. “已采取的即时行动”表述不够客观

如果当前代码尚未开始改动，那么文档中“已采取的即时行动”这一节会显得过满。

问题不在于方向，而在于措辞会让人自然理解为：

- 已经基于本次评审立即落地了一批修复
- 这些修复已经进入代码而非停留在意图层

在独立第三方视角下，这种写法不够严谨。

更稳妥的替代表述建议：

- 将“已采取的即时行动”改为“已确认的整改方向与现状说明”
- 将“已完成”改为“已识别 / 已确认 / 已计划 / 待落代码”

如果其中部分动作其实是此前已完成的历史改动，也应明确写成：

“该项为既有改动，不属于本次评审后的新增整改动作。”

### 2. “LLM 层已全面可用”表述偏满

回复中关于 LLM 能力的这类表述，需要更保守一些。

原因很简单：

- 当前代码确实支持 OpenAI-compatible 风格调用
- 但“支持兼容协议”不等于“所有列出的 LLM 服务商都已被真实账号、真实模型、真实错误语义充分验证”

因此更专业的说法应是：

- “大多数 LLM 服务商具备通用接入能力”
- “理论上可用或大概率可用”
- “仍建议逐服务商做真实端到端验证”

而不是直接写成“已全面可用”。

### 3. Gateway 风险部分虽正确，但缓和语气略重

回复中提到 `/report` 和 `/redeem` 尚未被客户端调用，这能解释优先级，但不能实质消除风险。

因为从第三方审视角度，风险判断依据是：

- 端点是否存在
- 语义是否自洽
- 并发与幂等是否成立

而不是“当前主客户端有没有调用到”

所以更客观的表达建议是：

“风险成立，但当前实际暴露面和业务使用度有限，因此优先级可与付费功能落地节奏协同处理。”

## 四、是否全面

这份回复在“覆盖原评审意见”这件事上，整体是全面的，已经覆盖了我前一份评审中的主要结论。

但如果要进一步提高专业性，我建议补两点：

### 1. 区分“现状承认”和“整改承诺”

当前文本里这两类内容有些混在一起。更好的结构应该是：

- 现状是否属实
- 风险级别如何判断
- 当前代码是否已修
- 后续准备怎么修

这样外部读者更容易分辨“已经发生了什么”和“接下来准备做什么”。

### 2. 给出“完成判定标准”

例如：

- 凭证迁移完成的标准是什么
- LLM 日志治理完成的标准是什么
- Provider 能力对齐完成的标准是什么
- Gateway 版本一致性完成的标准是什么

这会让后续复核更客观，也更容易验证。

## 五、统一口径建议

我建议开发团队先把回复文档统一到以下口径，再开始实际代码改动：

1. 保留对核心问题的采纳结论，不必弱化。
2. 删除或收敛一切会让人误解为“代码已修复”的表述。
3. 将“LLM 层已全面可用”改为“LLM 层大多已具备通用接入能力，仍需逐服务商实测验证”。
4. 将“已采取的即时行动”改为“已确认的整改项与现状说明”。
5. 将每个整改项补上可复核的完成标准。

## 六、最终意见

我的最终意见如下：

- 这份回复是合格的，整体专业、诚实、覆盖面足够。
- 我认可其大多数技术判断与优先级安排。
- 我不完全认可其中两类偏满的措辞：
  - 将未落代码的事项写得像已完成动作
  - 将兼容接入能力写成全面可用能力

因此，我建议：

**先修正文档口径，再开始代码整改。**

等代码真正改完后，我可以基于实际改动再次做独立复核，而不是只对计划做评价。
