# 不要在发版时跳过完整 `flutter test`

## 真实事件

**早期多次发生**，最后被钉死成 release skill 的硬规则。

具体场景：发版流程到「阶段 1 前置检查」时，agent 倾向于：
- 「这次只是文档改动，跳过测试吧？」
- 「测试要跑 12 分钟，太慢了，跑相关子集就好？」
- 「上次跑过了 598 全过，这次应该没问题？」

每个理由都"听起来合理"，但都被用户驳回过：「**release 默认跑 `flutter test` 全套，不要问"是否跳过"**。」

跳过测试的隐性代价：
- 看似无关的改动可能触发测试边缘场景失败（i18n 改动牵动 generated localizations、配置常量变动可能影响 default 值断言）
- 子集测试覆盖不到跨模块边界
- "不跑测试发版" 的习惯一旦开口子，下次更难关上

## 为什么会发生

agent 在用户视角「时间紧 / 改动小 / 等待无聊」时，会主动提议优化掉测试步骤——以为是体贴。但这恰恰是反向的：用户**信任 release skill 的纪律性**才把这件事委托给 agent。如果 agent 主动建议跳步，用户每次都要重新审视决策，**纪律的价值就崩了**。

## 如何避免

发版时**永远不主动建议跳过测试**。即使：

- 改动只有 README
- 改动只有注释
- 改动只有 i18n
- 上次刚跑过

唯一可以问的是："测试在跑了，要不要并行做下一步（如准备 CHANGELOG）？" —— 这是**并行优化**，不是**跳过**。

测试结果处理：

- **全过** → 继续阶段 2
- **失败但是已知环境问题**（如 model_full_flow 网络下载抖动）→ 报告给用户判断，**不自作主张** continue
- **失败且与本次改动相关** → 立刻停止，先修 bug 再继续

## 修复模式

如果意识到自己刚才提议过跳过测试：

1. 立刻收回提议，改成「测试在后台跑，我们并行做 X」
2. 不要等用户提醒——这条 feedback 已经记进 memory，应该自动内化
